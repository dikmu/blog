当B/S架构遇见分布式内存计算Coherence








B/S架构和存在的问题

“无边落木萧萧下，不尽长江滚滚来”，话说信息时代，IT风云际会，浪潮汹涌，持续推动着技术形态的不断变化。在现在企业的应用当中，大部分的架构都是采用B/S架构实现。始于1995年java 和miscosft visual studio 95的发布以及Windows95当中浏览器成为一种标配，直接推动了B/S架构的大行其道，至今已经有20余年的历史，B/S架构成为一种当然的选择和事实上的标准。

B/S架构在确定的用户数与确定的压力之下，没有太大的问题，然而大量的业务系统在不确定的压力和大并发压力情况下，系统性能满足不了系统的要求。主要问题如下：

  • 整体应用的性能受限于数据库的处理能力，采用此种架构的整体瓶颈一般在于数据库端，数据库中的数据量，随着时间的积累和业务的发展，数据库库中的表会越来越多，表中的数据量也会越来越大，相应地，数据操作，增删改查的资源开销也会越来越大；另外，由于无法进行分布式式部署，而一台服务器的资源（CPU、磁盘、内存、IO等）是有限的，最终数据库所能承载的数据量、数据处理能力都将遭遇瓶颈。

  • 业务逻辑与数据紧耦合，业务逻辑通过存储过程实现，升级优化复杂。

  • 争抢资源，影响关键业务，数据库压力过大，大量的查询业务（占整个系统的1/2以上），而关键的事务性业务处理常常面临资源不足和响应速度下降的困扰。

现在的解决方式

性能瓶颈既然在数据库，那就提升数据库的性能好了，通常只能采用垂直扩展的方式，即采用硬件扩容的方式，增加CPU和内存数量，实现更多的处理能力，采用此种方案，可行不，当然是可行的，高端的小型机和一体机和高端的存储，可以解决当前的性能问题，那是土豪的解决方案，一般的企业机构，真心是采用不起啊，而且未来业务再遇到瓶颈，怎么办，不能没完没了的升硬件吧。

既然一台数据库服务能力有限，能否把数据分布在多台机器上，是一个自然的想法和选择，于是乎“分库分表”的方案就隆重出炉了。可是数据如何分的合理，性能瓶颈在什么地方需要考虑。同时分库带来的新的问题是在执行分库分表之后，由于数据存储到了不同的库上，数据库事务管理出现了困难。分库分表后，额外的数据管理负担和数据运算压力也将大大增加。虽然解决了部分问题，维护量却上来了，一句话，人工的分布式，真心的累啊。

基于内存分布式缓存是个好的解药

 “众里寻他千百度，蓦然回首，那人却在，灯火阑珊处”，正当为解决性能苦苦挣扎之时，架构遇见基于内存的分布式计算缓存，遇见了Coherence，于是一切都改变了，问题不再是问题了。通过采用分布式缓存，将B/S架构的由集中式变为分布式架构。采用缓存的系统架构如下：


在这样的架构下，缓存服务器承载了交易所需要的数据，业务的交易围绕着缓存开展，而数据库服务器承担存储和事务保障，同时将大量的查询负载转移到了缓存当中，通过引入缓存，即时是相同的数据库服务器，系统的处理性能也大大提升。

什么是分布式计算呢，引百度的解释“分布式处理则是将不同地点的，或具有不同功能的，或拥有不同数据的多台计算机通过通信网络连接起来，在控制系统的统一管理控制下，协调地完成大规模信息处理任务的计算机系统。”哪什么事分布式缓存呢，引用百度说明，“缓存这种能够提升指令和数据读取速度的特性，随着本地计算机系统向分布式系统的扩展，在分布式计算领域中得到了广泛的应用，称为分布式缓存”。通俗的讲，实现多台计算机内存统一调度管理使用的数据容器。

引入分布式缓存，除了高性能这个看得到的特性外，应当说一个好的分布式缓存应当具有的特性如下：

  1. 在线的伸缩性：集群成员可以在线的增加或者减少，数据会自动进行分布，而且不影响正在的开展业务。

  2. 无单点故障：机器出现宕机，不会导致数据丢失，也不能影响业务运行。

  3. 高可靠性：数据可采用不同的策略进行备份，分布在不同的节点上，该节点可分配在不同机架不同的硬件服务器上。甚至实现跨数据中心的数据分布。

Coherence有那么高大上吗

说到分布式缓存，coherence是当然的No.1，虽然这个词儿大家伙儿听着很陌生，也很少听人讲起，但实际上，如果你用过weblogic，你应当就会见过它，只不过不认识，不了解而已。 Coherence是2001年发布的第一个版本，到现在已经15年的历史，历经演进，目前最新的版本为12.2.1，与开源的产品相比，coherence是一个出生在时代前沿的产品，是一个久经考验和稳定的产品。

  1. 自动化管理：集群和数据自动化管理，无须无主控节点和人工干预，coherence集群自动管理，无须人工干预，coherence的数据自动分布，访问透明。可以在线伸缩集群，无须停机。。

  2. 适应性强：X86服务器即可运行，即时在虚拟化的系统上也跑的妥妥的。想现在的服务器上运行，各家小型机上也没问题；甚至IBM的大机上也可以运行。而且经过了官方的认证。也就是说,coherence这个产品，是上得了厅堂，下得了厨房。

  3. 小而美：没错，就是想简单，软件包小一点，能绿色安装，云化安装。我想说的是，coherence运行，仅需要一个java和coherencer.jar，即可运行起来，最新版本的coherence.jar仅仅14.5兆。

  4. 想要搞应用级双活：coherence的联邦集群，实现跨数据中心的分发机制，可以实现从从双活、主备、分发，集中等数据发布模式，当然了你也可以自己定义你想要的数据分发模式。更当然的是，这是配置化实现的哦，简单吧，亲。

  5. 一个集群中的数据是否可以跨集群导出导入？可以，coherence集群中的数据可以导出到另外一个集群当中。不必再从数据库中导入。

  6. 有没有基于缓存的队列？有，如果你用了jms，但是被大数据量和高并发所困扰，jms on coherence是你理想的选择，而且这部分的代码是开源的哦。

  7. 我的系统是非JAVA平台的（C++，.net）的，这个能用coherence吗？很不幸的是，coherence不仅提供了C++和.net的接口，你完全可以使用，而且你只想访问数据，不想管数据如何存储的，coherence还提供了rest的接口。当然了如果会使用memcached ，coherence也提供了相应的接口。

Coherence与最新的技术融合特性?

你可能会有一定的疑问，coherence虽然很好，但对最新的技术支持怎么样，可以负责任的讲，Coherence对最新技术的支持是比较到位和即时的，而且也比较开放。请看:

  1. Coherence的云能力

    Coherence已经正式在oracle公有云发布，在oracle的公有云站点（cloud.oracle.com），java cloud server当中可以很方便订阅到coherence公有云服务。而且在oracle共有会上，提供针对coherence的rest api接口，实现增加、扩展、删除以及查看coherence节点。

    而且coherence也实现了多租户的管理，通过WLS,coherenc具有了多租户的特性，你可以根据需要进行资源的分配。

  2. docker容器的支持

    coherence支持Docker的技术，coherence的images可以在https://github.com/oracle/docker-images/tree/master/OracleCoherence找到。

  3. 对Java 8特性的支持

    Java8新增的Lambdas 和streams功能，coherence 已经全面支持。

  4. 不会使用coherence怎么办？

    途径有很多，可以联系oracle寻求技术支持，上oracle university进行学习等。互联网也是一个很好的途径，在https://github.com/coherence-community/coherence-demo提供了demo，供你参考学习。

  5. Coherence产品功能不支持的需求怎么办？

    Coherence发布出来的产品功能如果没有包含你所希望的功能，请你选别冲动自己开发解决问题，https://java.net/projects/cohinc提供了一个开源的项目Incubator，目前里面包含了一些有用，但使用相对不是很广泛的功能。比如基于cohernece的jms，cohrence 监控的jvisual代码等等。

如果你还需要其他的资源，可以登录https://coherence.java.net/，上面有更多的学习资源和与coherence相关的资源。

Coherence可以适用的业务场景有哪些？

Coherence发展到现在，我们可以这么说，数据库可以完成的工作，coherence基本都可以做，比如事务处理，BI分析等。数据库做不到的实时事件处理（比如秒杀业务），大企业日志采集等，也是coherence的得心应手的特性之一，下面仅仅就一些典型的业务场景进行一些说明：

  1.  数据高速缓存

    通过预先缓存数据源的数据（例如配置参数、机构信息、用户个人信息等），当应用需要数据时，应用程序直接从Coherence里面取得而不需要从后台数据源取得，可以极大降低对后台数据源的压力，极端情况下当后台数据源不可用的时候也不影响应用系统的可用性。

  2. 高并发查询

    通过在Coherence内存数据网格中共享应用对象和数据，可以实现高并发查询（例如二代支付网银批量交易状态同步查询），支持使用多个节点的并发查询，极速返回查询结果，大大提高客户满意度。

  3. 快速交易

    Coherence内存数据网格具有数据存储的高可靠性、完整性和安全性，在Coherence里可以直接完成交易处理，从而得到极快的响应速度。

  4. 事件实时响应

    Coherence内存数据网格可以使用事件驱动的应用架构，能够对事件做出实时的处理，使一系列应用需求可以高效实现，从提高服务质量和客户满意度。

  5. 实际使用场景介绍

    说了那么多分布式缓存和IT架构的改变，有没有在国内落地的案例，别忽悠人啊。真心来讲，coherence在国外用的较多，

总结

“金鳞岂是池中物，一遇风云便化龙。九霄龙吟惊天变，风云际会浅水游。”，传统的B/S架构，不仅面临着性能的瓶颈，而且也面临着云化需求。通过缓存优化架构，一切终将不同。通过分布式缓存来优化应用系统架构，可以提升通过水平扩展提升系统的性能。而且不在受限于你所使用的平台的操作系统，不受限于你说使用的开发工具（java、C++、.net），也不仅仅可以部署你的数据中心，你也可以选择公有云的模式使用coherence。数据也不再被冻结固定在数据库服务器上，而是数据变成了水，在服务器上流动。一句话，coherence是你建设新一代系统、优化改造老系统的利器。
