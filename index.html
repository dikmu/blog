基于RBAC的安全系统用例图

RBAC（Role-Based Access Control，基于角色的访问控制），就是用户通过角色与权限进行关联。简单地说，一个用户拥有若干角色，每一个角色拥有若干权限。这样，就构造成“用户-角色-权限”的授权模型。在这种模型中，用户与角色之间，角色与权限之间，一般者是多对多的关系。

角色是什么？可以理解为一定数量的权限的集合，权限的载体。例如：一个论坛系统，“超级管理员”、“版主”都是角色。版主可管理版内的帖子、可管理版内的用户等，这些是权限。要给某个用户授予这些权限，不需要直接将权限授予用户，可将“版主”这个角色赋予该用户。

当用户的数量非常大时，要给系统每个用户逐一授权（授角色），是件非常烦琐的事情。这时，就需要给用户分组，每个用户组内有多个用户。除了可给用户授权外，还可以给用户组授权。这样一来，用户拥有的所有权限，就是用户个人拥有的权限与该用户所在用户组拥有的权限之和。

用例图

在笔者接触的大量案例中，大部分系统是可采用为每个用户/角色逐一授权的形式进行开发的。由于授权信息完全记录在数据库中，在进行数据读取时就会混入大量的权限代码，导致应用效率地下。这其中不乏华为这样的大企业，其内网也是使用这种挨个授权的方式。

笔者也曾尝试过在自己主导的系统中使用更先进的RBAC授权方式，但无奈传统授权方式在业务、需求人员的思想中已根深蒂固。

——muhongdi@qq.com / blog.zhucai.me

当B/S架构遇见分布式内存计算Coherence








B/S架构和存在的问题

“无边落木萧萧下，不尽长江滚滚来”，话说信息时代，IT风云际会，浪潮汹涌，持续推动着技术形态的不断变化。在现在企业的应用当中，大部分的架构都是采用B/S架构实现。始于1995年java 和miscosft visual studio 95的发布以及Windows95当中浏览器成为一种标配，直接推动了B/S架构的大行其道，至今已经有20余年的历史，B/S架构成为一种当然的选择和事实上的标准。

B/S架构的服务器端典型部署框架如下：


B/S架构在确定的用户数与确定的压力之下，没有太大的问题，然而大量的业务系统在不确定的压力和大并发压力情况下，系统性能满足不了系统的要求。主要问题如下：

  • 整体应用的性能受限于数据库的处理能力，采用此种架构的整体瓶颈一般在于数据库端，数据库中的数据量，随着时间的积累和业务的发展，数据库库中的表会越来越多，表中的数据量也会越来越大，相应地，数据操作，增删改查的资源开销也会越来越大；另外，由于无法进行分布式式部署，而一台服务器的资源（CPU、磁盘、内存、IO等）是有限的，最终数据库所能承载的数据量、数据处理能力都将遭遇瓶颈。

  • 业务逻辑与数据紧耦合，业务逻辑通过存储过程实现，升级优化复杂。

  • 争抢资源，影响关键业务，数据库压力过大，大量的查询业务（占整个系统的1/2以上），而关键的事务性业务处理常常面临资源不足和响应速度下降的困扰。

现在的解决方式

性能瓶颈既然在数据库，那就提升数据库的性能好了，通常只能采用垂直扩展的方式，即采用硬件扩容的方式，增加CPU和内存数量，实现更多的处理能力，采用此种方案，可行不，当然是可行的，高端的小型机和一体机和高端的存储，可以解决当前的性能问题，那是土豪的解决方案，一般的企业机构，真心是采用不起啊，而且未来业务再遇到瓶颈，怎么办，不能没完没了的升硬件吧。

既然一台数据库服务能力有限，能否把数据分布在多台机器上，是一个自然的想法和选择，于是乎“分库分表”的方案就隆重出炉了。可是数据如何分的合理，性能瓶颈在什么地方需要考虑。同时分库带来的新的问题是在执行分库分表之后，由于数据存储到了不同的库上，数据库事务管理出现了困难。分库分表后，额外的数据管理负担和数据运算压力也将大大增加。虽然解决了部分问题，维护量却上来了，一句话，人工的分布式，真心的累啊。

基于内存分布式缓存是个好的解药

 “众里寻他千百度，蓦然回首，那人却在，灯火阑珊处”，正当为解决性能苦苦挣扎之时，架构遇见基于内存的分布式计算缓存，遇见了Coherence，于是一切都改变了，问题不再是问题了。通过采用分布式缓存，将B/S架构的由集中式变为分布式架构。采用缓存的系统架构如下：


在这样的架构下，缓存服务器承载了交易所需要的数据，业务的交易围绕着缓存开展，而数据库服务器承担存储和事务保障，同时将大量的查询负载转移到了缓存当中，通过引入缓存，即时是相同的数据库服务器，系统的处理性能也大大提升。

什么是分布式计算呢，引百度的解释“分布式处理则是将不同地点的，或具有不同功能的，或拥有不同数据的多台计算机通过通信网络连接起来，在控制系统的统一管理控制下，协调地完成大规模信息处理任务的计算机系统。”哪什么事分布式缓存呢，引用百度说明，“缓存这种能够提升指令和数据读取速度的特性，随着本地计算机系统向分布式系统的扩展，在分布式计算领域中得到了广泛的应用，称为分布式缓存”。通俗的讲，实现多台计算机内存统一调度管理使用的数据容器。

引入分布式缓存，除了高性能这个看得到的特性外，应当说一个好的分布式缓存应当具有的特性如下：

  1. 在线的伸缩性：集群成员可以在线的增加或者减少，数据会自动进行分布，而且不影响正在的开展业务。

  2. 无单点故障：机器出现宕机，不会导致数据丢失，也不能影响业务运行。

  3. 高可靠性：数据可采用不同的策略进行备份，分布在不同的节点上，该节点可分配在不同机架不同的硬件服务器上。甚至实现跨数据中心的数据分布。

Coherence有那么高大上吗

说到分布式缓存，coherence是当然的No.1，虽然这个词儿大家伙儿听着很陌生，也很少听人讲起，但实际上，如果你用过weblogic，你应当就会见过它，只不过不认识，不了解而已。 Coherence是2001年发布的第一个版本，到现在已经15年的历史，历经演进，目前最新的版本为12.2.1，与开源的产品相比，coherence是一个出生在时代前沿的产品，是一个久经考验和稳定的产品。

  1. 自动化管理：集群和数据自动化管理，无须无主控节点和人工干预，coherence集群自动管理，无须人工干预，coherence的数据自动分布，访问透明。可以在线伸缩集群，无须停机。。

  2. 适应性强：X86服务器即可运行，即时在虚拟化的系统上也跑的妥妥的。想现在的服务器上运行，各家小型机上也没问题；甚至IBM的大机上也可以运行。而且经过了官方的认证。也就是说,coherence这个产品，是上得了厅堂，下得了厨房。

  3. 小而美：没错，就是想简单，软件包小一点，能绿色安装，云化安装。我想说的是，coherence运行，仅需要一个java和coherencer.jar，即可运行起来，最新版本的coherence.jar仅仅14.5兆。

  4. 想要搞应用级双活：coherence的联邦集群，实现跨数据中心的分发机制，可以实现从从双活、主备、分发，集中等数据发布模式，当然了你也可以自己定义你想要的数据分发模式。更当然的是，这是配置化实现的哦，简单吧，亲。

  5. 一个集群中的数据是否可以跨集群导出导入？可以，coherence集群中的数据可以导出到另外一个集群当中。不必再从数据库中导入。

  6. 有没有基于缓存的队列？有，如果你用了jms，但是被大数据量和高并发所困扰，jms on coherence是你理想的选择，而且这部分的代码是开源的哦。

  7. 我的系统是非JAVA平台的（C++，.net）的，这个能用coherence吗？很不幸的是，coherence不仅提供了C++和.net的接口，你完全可以使用，而且你只想访问数据，不想管数据如何存储的，coherence还提供了rest的接口。当然了如果会使用memcached ，coherence也提供了相应的接口。

Coherence与最新的技术融合特性?

你可能会有一定的疑问，coherence虽然很好，但对最新的技术支持怎么样，可以负责任的讲，Coherence对最新技术的支持是比较到位和即时的，而且也比较开放。请看:

  1. Coherence的云能力

    Coherence已经正式在oracle公有云发布，在oracle的公有云站点（cloud.oracle.com），java cloud server当中可以很方便订阅到coherence公有云服务。而且在oracle共有会上，提供针对coherence的rest api接口，实现增加、扩展、删除以及查看coherence节点。

    而且coherence也实现了多租户的管理，通过WLS,coherenc具有了多租户的特性，你可以根据需要进行资源的分配。

  2. docker容器的支持

    coherence支持Docker的技术，coherence的images可以在https://github.com/oracle/docker-images/tree/master/OracleCoherence找到。

  3. 对Java 8特性的支持

    Java8新增的Lambdas 和streams功能，coherence 已经全面支持。

  4. 不会使用coherence怎么办？

    途径有很多，可以联系oracle寻求技术支持，上oracle university进行学习等。互联网也是一个很好的途径，在https://github.com/coherence-community/coherence-demo提供了demo，供你参考学习。

  5. Coherence产品功能不支持的需求怎么办？

    Coherence发布出来的产品功能如果没有包含你所希望的功能，请你选别冲动自己开发解决问题，https://java.net/projects/cohinc提供了一个开源的项目Incubator，目前里面包含了一些有用，但使用相对不是很广泛的功能。比如基于cohernece的jms，cohrence 监控的jvisual代码等等。

如果你还需要其他的资源，可以登录https://coherence.java.net/，上面有更多的学习资源和与coherence相关的资源。

Coherence可以适用的业务场景有哪些？

Coherence发展到现在，我们可以这么说，数据库可以完成的工作，coherence基本都可以做，比如事务处理，BI分析等。数据库做不到的实时事件处理（比如秒杀业务），大企业日志采集等，也是coherence的得心应手的特性之一，下面仅仅就一些典型的业务场景进行一些说明：

  1.  数据高速缓存

    通过预先缓存数据源的数据（例如配置参数、机构信息、用户个人信息等），当应用需要数据时，应用程序直接从Coherence里面取得而不需要从后台数据源取得，可以极大降低对后台数据源的压力，极端情况下当后台数据源不可用的时候也不影响应用系统的可用性。

  2. 高并发查询

    通过在Coherence内存数据网格中共享应用对象和数据，可以实现高并发查询（例如二代支付网银批量交易状态同步查询），支持使用多个节点的并发查询，极速返回查询结果，大大提高客户满意度。

  3. 快速交易

    Coherence内存数据网格具有数据存储的高可靠性、完整性和安全性，在Coherence里可以直接完成交易处理，从而得到极快的响应速度。

  4. 事件实时响应

    Coherence内存数据网格可以使用事件驱动的应用架构，能够对事件做出实时的处理，使一系列应用需求可以高效实现，从提高服务质量和客户满意度。

  5. 实际使用场景介绍

    说了那么多分布式缓存和IT架构的改变，有没有在国内落地的案例，别忽悠人啊。真心来讲，coherence在国外用的较多，

总结

“金鳞岂是池中物，一遇风云便化龙。九霄龙吟惊天变，风云际会浅水游。”，传统的B/S架构，不仅面临着性能的瓶颈，而且也面临着云化需求。通过缓存优化架构，一切终将不同。通过分布式缓存来优化应用系统架构，可以提升通过水平扩展提升系统的性能。而且不在受限于你所使用的平台的操作系统，不受限于你说使用的开发工具（java、C++、.net），也不仅仅可以部署你的数据中心，你也可以选择公有云的模式使用coherence。数据也不再被冻结固定在数据库服务器上，而是数据变成了水，在服务器上流动。一句话，coherence是你建设新一代系统、优化改造老系统的利器。

Oracle SOA 12c加速面向服务架构实现








面向设计服务架构可以用于创建灵活的、可管理的企业系统，但是技术及消费化的新趋势已经改变了SOA系统的需求。如果使用面向服务架构的组织希望维持当前继续满足用户和利益相关者需求的系统，那么面向SOA的架构必须满足移动和社交媒体方面的要求，与此同时也能够适应和融合云技术。

然而面对云时代，移动互联，物联网的时代，企业与架构师在对待SOA时往往有存在一些误解，由于这些误解的存在，企业内面相服务的架构与云，移动，物联网结合的时候，一些常见的绊脚石往往被过度扩大了。一旦理清了SOA的这些误解，再借助于业内主流提供商的SOA产品，就能发现，加速SOA的实现，能极大加速企业IT向移动互联网、物联网和云的过度。

误解一：SOA已趋于死亡，API、微服务才是未来

事实上，无论是微服务论还是REST论的争议由来已久，但主流的声音还是认为无论是微服务，还是REST的出现，并不是用来取代SOA的。SOA将转向支持物连、移动化和云。人们将用它支持面向特定任务的服务，它们称为应用程序接口-APIs。这些API用于移动设备和云计算解决方案。一个典型例子是web页面上的“喜欢”或“发到朋友圈”按钮。这些系统通过API进行交互，现在，API常常作为SOA的“门面”，由SOA提供整合、安全性、治理和管理等服务。

误解二：SOA很复杂，不好用

这绝对是个谬论，SOA并不复杂。当SOA出现还是少数学院派口中的名次时，虽然思想上各家都能达成共识，但是各自的实现却不一致。加之一些产品仅仅是把多个功能部件简单整合，光是SOA产品本身的使用和管理就有多种组合和多种工具，这一阴影在相当长一段时间内是架构师和开发者挥之不去，甚至时至今日，还在影响他们的选择。事实上，成熟的SOA产品会提供简单有效的工具，通过服务的编排，整形、路由，策略等实现复杂应用和业务流程的组合，以提供更好的敏捷性。甲骨文SOA提供了单一的安装包和全可视化开发工具JDeveloper以及DevOps 的支持，30分钟即可实现从零开始的环境部署，这已经达到和接近于PaaS和SaaS的部署复杂度。额外的，甲骨文SOA云服务也可实现传统SOA服务向云端的一键切换。这么一说，SOA远没想象的复杂，不是吗？问题的关键是要选对产品和方案。

误解三：SOA仅仅一项技术

SOA对于企业IT来说，这意味着一种技术架构，意味着服务开发设计的思想。事实上SOA的软件提供商正是从设计架构角度，借助于成熟工具，封装并屏蔽晦涩的技术。通常现代化的程序技术发展如移动化、物联网等等都会带来很多新的技术。比如大数据听起来似乎和SOA没啥关系，但是低延迟、实时通讯、复杂事件处理、事务处理等都已经在 SOA中实现，通过分析并针对事件运用规则的方式解决大数据问题。所以，无论是当前前沿科技还是传统技术，甲骨文SOA中“包容兼蓄，共生共荣”的特性能够稳定为企业输出灵活IT架构，制成未来业务发展转型。SOA中的服务梳理和分析，服务流程优化，服务管控和治理都将成为有效支撑企业服务架构。相比无论是微服务，还是REST，SOA侧重点远不是单一的技术层面，而是创建以新技术创新为驱动，稳健基础平台为支撑的全新的未来企业IT架构。

既然SOA是设计原则，那么它又饱含哪些内容与更新呢。事实上，甲骨文公司一直致力于在模糊性和宽泛性的SOA上总结原则与方法，归纳并标准化。在如今客户机－服务端模式逐渐向互联网、云模式变革演进过程中，认清方向，抓住本质。在掌握和了解宏观SOA概念同时，我们更要着眼于SOA落地，那我们我们来看看甲骨文SOA套件如何更加贴近实际、贴近用户。

Docker+甲骨文中间件—为企业应用插上翅膀








企业需要 Docker ，企业应用需要Docker。尽管非常年轻， Docker 已被列在诸多企业的技术路线图上，成为金融、保险、电信、电商、游戏、初创企业的心头所好。2014年开始，云计算领域的宠儿OpenStack似乎被一个新名词取代了：Docker, 它是一个开源的应用容器，可以方便的将你开发的应用以及相关的依赖包打包到一个容器里面，秒级启动你的应用，支持你创建可扩展的、轻量级的环境来承载你的应用，更支持你通过层次化的方案对开发的应用和产品进行管理与维护。

就像硬币的两面，也有一些企业认为没有必要使用 Docker 而把自己转变为DevOps布道者的乐土；他们不需要微服务架构，也无需敏捷的全栈工程师，但事实上， Docker 非常适合这些传统企业的IT转型，能够帮助团队实施开发运维一体化的最佳实践。从企业应用的角度，Docker可以从以下三个方面使应用交付更加畅快：

  • DevOps：对于DevOps方法论最佳实践来说，以前虽然有很多技术出现，但是自从Docker出现之后，它更加贴近研发和运营。以前可能Docker更多的实践是在运营团队，现在它可以保证线上的环境，保证更多的一致性。每一个Docker的服务可能是无状态，其中一个容器出现了问题，再用一个容器来替代。基于当前的服务的概念，其他容器管理技术来说，无状态就是更加的常态化。

  • 持续集成：持续交付是软件交付和应用交付比较注重的概念，Docker的出现使得持续集成被大家广泛的接受。基于一些现有的技术与平台，它可以使你的开发和测试更加快速。如果我们将Docker的概念结合起立，其实每一个Docker的环境构建，每一个版本的迭代更加快速。还有线上线下的环境是一致的，每一个应用可以在不同的环境里，保证环境的统一，减少了我们以往做的很多测试。

  • 协作：Docker还可以让我们在不同的领域之间协作。我们现在有Docker Hub以及本地部署的Docker Registry，它可以扩展，比如说需要一个服务或者应用或者产品，可能有在这个领域比较擅长的公司、擅长的团队有一些好的实践，我们可以与其他的公司或者团队展开协作，以这个为基础做我们自己的服务，专注我们自己擅长的领域，更加关心属于自己的业务以及所带给客户的价值。

Docker和容器技术如今备受媒体关注，这种强大而简易的方法可用来封装应用程序，尤其是由于云计算变得更主流。虽然这类技术仍在发展之中，但它们提供了一种简单、干净又精简的方式来分配应用程序工作负载。

大多数企业已经开始使用开发运维一体化的工具链，不过这些过程都比较混乱。企业们纷纷雇用云计算顾问、强化数据中心、打破工程师团队之间的隔阂，以及将应用迁移到云上，尽管迁移到云具有灵活性等优点，跨多个复杂系统进行安全和扩展性管理仍是一项工程学壮举。大部分的企业应用饱受内部依赖、网络复杂性、庞大的定制化数据库集群等的困扰。

Docker+甲骨文中间件可以弥合这一鸿沟。企业使用 Docker 的首要原因就是帮助他们跨系统部署、迁移应用、以及消除人工重复配置的工作。由于应用的依赖内建于容器， Docker 容器能够大幅减少对互用性的焦虑。 Docker 在裸机服务器、本地虚拟机、 云实例上都能运行良好,但是仅有Docker，对于企业应用来说远远不够，企业可以借助甲骨文中间件提高应用的性能和可扩展性。

其中，Weblogic Server 12c是甲骨文在2015 开放日上宣布推出，该产品是10年来规模最大的Oracle WebLogic Server版本，也是全球首款云原生、企业级Java平台。除了支持多重租用和多个高可用性数据中心，新版本还全面支持Java EE 7和Java SE 8，企业还可以通过基于Oracle WebLogic Server的Oracle Java云服务，在本地和云数据中心中使用同一开发平台。

微服务架构探讨及甲骨文中间件微服务技术解决方案








随着传统企业受到互联网+的冲击，越来越多的企业都在面临业务转型，如何更好地贴近客户以获取更高的客户满意度，如何在企业内部加速供给侧改革，实现更好的供需平衡都是企业在业务转型中需要思考的问题。企业业务转型，离不开底层IT架构的支撑，所以最近很多很火的技术理念不断被大家所谈及，包括微服务架构、DevOps开发运维模式、数据集装箱Docker技术等。甲骨文也是一家在技术和理念不断创新的公司，今天就其中的微服务架构主题展开，谈一下我们的看法及相关中间件产品支撑的解决方案。

谈及微服务架构，就不得不提及传统单体应用（monolithic）架构，如下图所示：


单体应用有以下几个特点：

  1. 单体应用的部署物（范围）一般是某个完整的业务应用场景，比如网银、社保或更大的像ERP等。

  2. 采取中心化的数据存储方式，虽然可以实现集群或数据分区，单数据存储始终是集中式的。

  3. 采用单一的技术框架或优化方式，例如整个单体应用都是采用统一的语言编写，例如Java, .Net等，所有的优化，包括CPU、内存、IO、网络资源都是基于单体应用完整实例进行。

  4. 扩展无论是垂直还是水平扩展的单位，也都是单体应用本身。

  5. 从开发运维模式来讲，基本都是开发和运营独立，人员组织机构基本上按照UI, 应用逻辑和数据的应用三层技术架构进行划分。

传统的绝大部分应用都是采用单体应用架构来进行设计，但是随着时间的推移，它的局限性也逐渐显现，主要包括以下几点：

１.  太复杂：

随着时间的推移，应用会变得越来越大，对开发者而言很难理解，而且共享层 (ORM, 消息处理等) 需应对100%的用户场景

２.  交付太慢：

团队按照功能划分– UI、应用、中间件和数据库等. 由于相互交叉，永远需要等所有的事情都做完才能交付

３.  太脆弱：

一个bug 会很快毁掉整个应用，缺乏弹性

４.  缺乏优化：

应用的不同组件有不同的需求–更多CPU、更多内存和更快的网络等，单体应用无法就某个功能组件进行优化。

５.  代码没有归属者：

代码将成为“公地悲剧”的受害者– 因为代码没有归属者，出了问题你只能选择忽略它

６.  无效率测试：

每次你改动应用，你必须重新测试整个应用，很难支持连续交付的需求。

微服务架构的出现基本上是对现有复杂大型单体应用架构以业务为单元进行拆分，每个拆分的微服务应用都可以单独部署和测试，可以采用单独的技术架构，独立的数据存储，独立的开发运营团队支撑，快速以微服务应用为单位进行弹性扩展，降低各个业务单元之间的耦合关系。具体和单体应用的对比可详见下图：


         如果企业出现了下面５个迹象，则可以尝试采用微服务架构进行应用设计：

  1. 一个应用需要100+ 开发者

  2. 一个应用＞5M代码行

  3. 以月或者季度为单位来发布应用到生产系统

  4. 积压 > 1 季度的开发工作

  5. > 20% 开发者的人事变动

从下图可以看出，通常来讲单体应用依然是更好的选择，对于简单和中等复杂程度的应用，无论是长期还是短期来看其成本开销都是好于微服务架构，对于非常复杂的应用, 微服务架构长期来看会有回报，但是需要经历很长时间来弥合前期的巨大投资。


通常客户会混淆SOA和微服务，因为他们之间有很多相似的地方，包括松耦合的架构，通过模块化来降低复杂度等相似思想。甲骨文认为SOA是一个通用的理念，而微服务是其中一种具体的实现方式。两者最大的区别在于实现方式，SOA更多的是基于ESB来实现服务编排，我们可以把它称之为“简单接入，智能管道”，而微服务更多的是强调去中心化，与之相对应的是“智能接入，简单管道”。更详细的对比详见下图：


接下来我们简单讲一下微服务的实现方式，从几个角度来看，一个是架构角度，一个是技术实现层面。架构方面由于每个微服务应用都有自己的数据存储（如下图所示），所以微服务导致架构向分布式计算迁移，这里就需要考虑几个问题，第一根据CAP原则，我们需要再一致性、分区容忍度及可用性之间寻求平衡，这会造成最终一致性问题，我们在设计架构的时候一定要能够提前评估。第二由于所有微服务间的数据交换都需要通过API层，或者消息的方式来实现通讯，而不是原来单体应用内线程调用，其通讯成本会变得很高，同步的调用方式并不是最佳选择，一般会采用异步的消息方式来进行。


技术层面的实现技术核心组件主要包括以下几方面：

  1. 服务注册：管理每个微服务接入点的全生命周期，自我感知多租户，微服务的版本和环境信息。

  2. API负载均衡器：匹配每个请求到最佳接入点，查询服务注册来决定最佳接入点，为每个HTTP请求寻找最佳接入点。

  3. API网关：为每一种类型（Web和移动端等）的客户端构建XML或者JSON的响应，异步的调用N个微服务来构建所需的响应，处理安全以及对后台的隐藏负载均衡，应用有限的应用逻辑。

  4. 安全性：必须认证和授权每个单独的主体，单体应用典型的都是自己来认证和授权，每一个单独的远程网络调用都要进行安全保护，可以利用网络分段来隔离各个微服务。

甲骨文中间件作为应用部署的最佳平台，对微服务的支持也是不遗余力，下面我们从几个方面来看看相关产品的支持：

１.  云服务支持：云服务的弹性及可扩展性是构建微服务架构很好的平台

  • Java云服务：基于J2EE的微服务应用可以直接部署在Java云中

  • 应用容器云服务：底层基于Docker的应用服务器云服务可以为微服务应用支持更多平台语言的选择，例如J2SE，Node.JS等

２.  WebLogic多租户：WebLogic的多租户特性可以实现应用的高密度部署，在共享资源的情况下，可以实现很好的隔离，同时可以支持基于分区方式的灵活迁移。这些特性对微服务架构都是必要的，我们知道微服务应用相对于原来单体应用拆分后，对高密度部署的要求更高，WebLogic的多租户特性可以保证在相同业务SLA的水平下降低服务器端 Java 基础架构的总拥有成本（硬件服务器数量减少66%，维护所需的工时数减少 25%）。


３.  WebLogic的Docker支持：Docker作为轻量的容器化技术，区别于传统的虚机的方式，可以实现更快的部署、启停以及迁移，这对于微服务的实现确实是有帮助的。WebLogic作为J2EE应用部署的最佳服务器，也在12.1.3及12.2.1的版本中增加了对Docker的认证和支持，可以基于单机或者多机的模式下进行基于Docker容器化方式的应用部署，这对于微服务应用基于Docker的实现可以说是一个更好的选择方式。

４.  Oracle API管理：前面我们介绍过，微服务的技术组成里包括了对API管理的相关功能，我们这里也简单介绍下相对应的甲骨文的中间件产品：

  • Oracle API Catalog：简化API的收集、注释和发布,易于应用开发者实现API的发现、消费、评价和评分

  • Oracle API Manager：API开发门户，发现和测试API，注册使用它们的应用程序，在运行时监控使用情况。

  • Oracle API Gateway：API安全网关，提供部署在本地或云端的应用访问的接入安全，提供数据格式转换和协议桥接。

最后我们来总结一下，微服务架构并不是一个银弹，他不能解决所有问题，我们也看到，没有一种开发模式，无论是在技术还是管理领域，可以承诺在10年内，无论是生产效率、可靠性还是简化程度可以领先其他技术一个数量级，所以我们需要根据实际的应用业务需求结合未来的发展趋势，做相应的抉择，选择最适合自己的软件架构，下图是我们就微服务应用的收益和成本的对比，作为最后的总结。
